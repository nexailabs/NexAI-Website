## Per-agent landing page gap

Six agents are advertised on the homepage rotation (`src/data/home.ts:7-14`) and roster (`HomeRoster.astro` derived from `agents` array). Only one has a dedicated page:

| Agent          | Has own page? | Current link target                   |
| -------------- | ------------- | ------------------------------------- |
| Outreach       | ❌            | rolls into `/coming-soon` (Agent Lab) |
| Marketing      | ❌            | rolls into `/coming-soon`             |
| CEO (Strategy) | ❌            | rolls into `/coming-soon`             |
| Finance        | ❌            | rolls into `/coming-soon`             |
| Research       | ❌            | rolls into `/coming-soon`             |
| Creatives      | ✅            | `/studio` (the only shipped agent)    |

**This is a deliberate strategy call, not a bug.** Per the design unification PR's locked decisions, Agent Lab launches with its own surface "once each one's been used by 10 teams in production" (verbatim from `coming-soon.astro:19-21`). The `/coming-soon` page exists exactly to absorb these clicks until then.

**Decision needed before scale:** when one agent (say, Outreach) ships, does it get `/agents/outreach` or `/outreach`? Both paths are unbuilt. Recommendation: `/agents/<slug>` to keep the namespace clean and free up top-level slugs for future products.

## Audit method

```
glob src/pages/**/*.astro          → enumerate routes that exist
glob public/**/*                   → enumerate static assets
grep "href=\"/"                    → enumerate internal links referenced from code
grep "href:\\s*['\"]/"             → enumerate internal links in TS data files
grep "id=\"(known anchor list)\""  → verify anchor targets
read public/robots.txt
read public/_redirects
read astro.config.mjs              → confirm sitemap integration
read Layout.astro                  → confirm OG / favicon refs
```

No conclusions drawn from convention alone — every "missing" claim was verified against the actual file tree.
